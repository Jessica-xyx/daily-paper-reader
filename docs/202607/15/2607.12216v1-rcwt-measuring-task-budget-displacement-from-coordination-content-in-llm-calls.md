## 摘要
多智能体和记忆增强的大语言模型系统通常将协调内容、共享状态、先前的讨论、工具输出、摘要和角色指令放在用于当前任务的同一有限提示中。这造成了一个实际分配问题：在固定上下文预算下组装调用时，用于协调的每个标记都无法用于任务指令或证据。我们引入了圆桌上下文窗口测试（RCWT），这是一种控制协议，用于测量这种任务预算位移效应。RCWT在控制总预算、位置顺序、任务族和评分的同时，改变协调内容。在W=4096的主要上下文相关召回任务中，三个商业模型在适度开销下保持在基线附近，然后当剩余参考证据降至几百个标记时急剧下降。窗口缩放摘要与特定于任务的任务剩余预算解释一致，而非固定百分比阈值，但我们将其视为描述性证据而非普遍规律。为了测试当任务证据保持完整时固定预算悬崖是否仍然存在，我们添加了一个完整任务消融实验：完整任务/参考块保持存在，同时通过扩展总提示长度来增加协调标记。在该设置中，所有测试的调用在GPT-4.1-mini、Claude Haiku 4.5和Gemini 2.5 Flash上，在高达95%的协调比例下，都正确返回了每个评分的字段。这一消融实验缩小了结论：主要的RCWT悬崖最好被解读为任务预算位移，而不是证明协调量本身在原始开放任务中引起语义干扰。因此，RCWT是上下文分配预算的一种测量原语，而不是多智能体效益或会话级别协调的完整理论。

## Abstract
Multi-agent and memory-augmented LLM systems often place coordination content, shared state, prior discussion, tool outputs, summaries, and role instructions, inside the same finite prompt used for the current task. This creates a practical allocation problem: every token spent on coordination is unavailable to task instructions or evidence when a call is assembled under a fixed context budget. We introduce the Roundtable Context Window Test (RCWT), a controlled protocol for measuring this task-budget displacement effect. RCWT varies coordination content while controlling total budget, position order, task family, and scoring. In the main context-dependent recall task at $W=4096$, three commercial models remain near baseline through moderate overhead and then degrade sharply once residual reference evidence falls to a few hundred tokens. Window-scaling summaries are consistent with a task-specific residual-budget interpretation rather than a fixed percentage threshold, but we treat this as descriptive evidence rather than a universal law. To test whether the fixed-budget cliff persists when task evidence remains intact, we add an intact-task ablation: the full task/reference block is kept present while coordination tokens increase by expanding total prompt length. In that setting, all tested calls return every scored field correctly across GPT-4.1-mini, Claude Haiku 4.5, and Gemini 2.5 Flash up to a 95\% coordination ratio. This ablation narrows the claim: the main RCWT cliff is best read as task-budget displacement, not as proof that coordination volume alone causes semantic interference in the original open-ended task. RCWT is therefore a measurement primitive for context-allocation budgeting, not a complete theory of multi-agent benefit or session-level coordination.

### 1. 论文的核心问题与整体含义

- **研究动机**：在多智能体或记忆增强的大语言模型（LLM）系统中，一次模型调用需要同时容纳任务指令、检索文档、记忆摘要、智能体历史信息、工具输出、角色提示、共享状态等多种内容，它们被塞入同一个固定大小的上下文提示中。这导致了一个实际分配问题：用于“协调”的每个 token 都会挤占原本可用于当前任务指令或证据的预算。
- **核心问题**：在固定上下文预算下，协调内容（coordinator content）对任务性能的影响到底有多大？这种性能下降是由于任务证据被物理截断（预算位移），还是由于协调内容本身对模型推理产生了语义干扰？
- **整体含义**：作者提出 **RCWT（圆桌上下文窗口测试）** 作为测量这种任务预算位移效应（task-budget displacement）的控制协议，并证明性能下降的主要原因是任务证据的物理流失而非语义干扰，从而为工程实践提供了预算分配的安全边界。

### 2. 方法论

- **核心思想**：在固定上下文预算 W 下，将提示分为三块：固定任务指令块 u、协调内容块 c、技术参考块 r。定义目标比例 q = c/(W − u)，实际协调份额 p = c/W。通过改变 q，观察模型在上下文相关的召回任务上的性能如何随 c 增加而下降。
- **关键技术细节**：
  - 使用 cl100k_base tokenizer 进行确定性 token 计数，以确保跨供应商的可重复性。
  - 协调内容为合成的多智能体共享上下文（角色、协议、历史消息、工具模式等）。
  - 参考块为技术软件规格说明，包含 8 个可评分的事实点。
  - 模型回答为开放式的结构分析，使用二值事实检查法官（LLM judge）评分。
  - 实验在固定预算（W=4096）下进行，q 取值：0%, 25%, 50%, 75%, 90%, 92%, 94%, 96%, 98%（对应 p 约 0%~90%）。
- **公式/算法（文字说明）**：
  - 定义性能函数 RM(c,W,T) ∈ [0,1]。
  - 拟合多种候选曲线（logistic、幂律、指数、二次、分段线性），主要用 logistic：RM = R0 / [1 + exp(k(c/W − p0))]。
  - 定义任务特定的剩余预算参数 θ = W(1 − p0)，用于描述临界点对应的任务保留 token 数。
- **消融实验**：为了区分预算位移与语义干扰，设计了 **完整任务消融实验（Intact-task ablation）**：保持任务/参考块完整不变，协调内容围绕其增加，总提示长度随之增长。此时若性能仍保持，则说明悬崖效应源于任务证据被截断而非协调内容的语义干扰。

### 3. 实验设计

- **数据集/场景**：
  - **主要任务**：上下文相关的技术规格召回任务（10 个事实，排除 2 个地板效应项后剩 8 个）。
  - **边界任务**：
    - 自包含算法任务：所有信息都在任务块中，协调内容完全无关。
    - 外部基准包：GSM8K、MMLU-Pro、DROP 各组成 10 个问题的包，作为压力测试。
- **Benchmark**：主要实验固定 W=4096；窗口缩放实验扩展到 W=8192 和 W=16384 以检验比例 vs. 绝对剩余预算。
- **对比方法**：
  - 三个商业模型：Gemini 2.0 Flash（部分实验用 Gemini 2.5 Flash）、GPT-4.1-mini、Claude Haiku 4.5（部分实验用 Claude Haiku 4.5 20251001）。
  - 未与任何其他协议或基线模型对比，因为 RCWT 是测量协议而非竞赛。
- **实验设置**：每种条件两种顺序（协调优先/理由优先），主任务每模型-目标对 40 次调用（20 次×2 顺序）。完整任务消融实验每条件 10 次调用（5 次×2 顺序），使用确定性 JSON 评分（8 个确切字段）。

- **未明确说明**：论文未提及 GPU 型号、数量、训练或推理时长、能耗等算力信息。仅提到使用公共 API 调用商业模型（Gemini、GPT、Claude），推测为商业 API 调用，具体算力由提供商负责。因此无法总结算力细节。

### 5. 实验数量与充分性

- **主实验数量**：3 个模型 × 9 个目标比例 × 40 次调用 = 1080 次调用（实际因部分比例后加，总调用的数量合理）。
- **窗口缩放实验**：W=8192 每模型 40 次/对，W=16384 每模型 20 次/对，共约 3×9×40+3×9×20 等，足够。
- **完整任务消融实验**：3 模型 × 5 比例 × 10 次 = 150 次调用，所有调用均正确，统计上可靠（Wilson 95% CI [0.722,1]）。
- **边界任务**：每个模型-包各 100 次调用（10 项×2 顺序×5 重复），覆盖多个任务类型。
- **充分性评价**：实验数量充分，控制变量良好（位置顺序、预算固定、比例扫描）。但存在一定局限：任务类型有限（主要是技术规格召回和少数标准基准），且未对多个独立随机任务进行样本重复，仅基于同一组事实的多次调用。因此外部有效性有限。
- **客观性与公平性**：采用 LLM judge 评分可能引入评分器偏差；完整任务消融改用确定性 JSON 评分提高了客观性，但也改变了任务难度和格式（从开放式分析变为精确字段提取），因此两个实验不能完全直接对比。整体上实验设计考虑了位置控制和比例扫描，比较公平。

### 6. 论文的主要结论与发现

- **主要发现**：在固定预算（W=4096）下，三个模型在中等开销下保持高准确率，但当剩余参考证据降至几百个 token 时（目标 q=90% 对应参考仅 376 token），准确率骤降 15~34 个百分点。这表明任务预算位移是关键因素。
- **窗口缩放支持任务特定剩余预算解释**：W=8192 时同样的 p 值对应更多剩余参考（786 token），下降较小，说明临界点并非固定比例，而是绝对剩余 token 数（θ 约 568~665）。
- **完整任务消融实验反驳语义干扰假说**：当任务证据完整并且总提示长度增长到 95% 协调比例时，所有模型在所有调用（150/150 次）中全部正确，未出现悬崖。说明协调内容本身不会在大规模上引起语义推理损伤（至少在提取式任务中）。
- **边界任务显示任务依赖性**：自包含算法任务稳定；外部基准包在任务被截断时性能下降，但某些模型在未截断时就已开始下降（如 Claude Haiku 4.5 在 GSM8K 包上 p=0.25 时从 0.6 降至 0.4），表明可能存在其他干扰因素。
- **工程含义**：系统应该测量任务族所需的“剩余任务预算”，而不是只看名义上下文大小或总提示长度。

### 7. 优点

- **清晰聚焦的测量协议**：将复杂的多智能体协调问题简化为单一调用中的预算分配问题，方便分析和复现。
- **控制变量严格**：固定预算、固定位置顺序、确定性 token 计数，排除了许多混杂因素。
- **消融实验设计巧妙**：通过保持任务完整来区分预算位移与语义干扰，使结论更有力。
- **窗口缩放实验增强解释**：验证了剩余绝对预算而非比例是关键。
- **结果具有实用价值**：为工程实践提供了具体的“剩余预算”安全边界建议（例如需保持几百 token 的保留证据）。
- **开源代码与模板**：便于其他研究者复现和扩展。

### 8. 不足与局限

- **任务覆盖狭窄**：主要实验基于单个技术规格召回任务，外部有效性有限。虽然增加了三个基准包，但每个包仅 10 个问题，且任务类型仍以问答/推理为主。
- **评估方法不一致**：主任务使用 LLM judge（可能引入评分偏差），消融实验改用确定性 JSON 评分（任务格式和难度不同），无法直接对比量化语义干扰效应的大小。
- **协调内容单一且存在主题重叠**：合成协调块与参考事实存在主题重叠（CRDT、Redis、WebSocket 等），可能干扰结果。
- **前缀截断导致事实顺序固定**：参考块的前缀截断按固定顺序移除事实，使得临界点部分反映事实布局而非任务内在性质。
- **未测量协调的收益**：仅测量成本侧的位移，未评估协调带来的质量提升（如分解、验证、工具使用等），因此不能直接得出“协调有害”的结论。
- **计算资源未报告**：无法评估实验的经济成本和可重复性细节。
- **统计推断局限性**：主任务的自举置信区间仅反映同一提示下的运行间变异，不代表对独立任务总体的不确定性。
- **模型版本变化**：部分实验使用了不同版本的 Gemini 和 Claude，可能引入细微差异。
- **多智能体动态缺失**：协议针对单次调用，未考虑会话级动态（如记忆编写、检索策略、工具故障等），实际多智能体系统可能更复杂。
