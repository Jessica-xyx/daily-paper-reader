## 摘要
长视野LLM智能体会积累工具结果、文件、计划以及用户约束，这些内容结构复杂，不能简单地作为一次性文本后缀处理。当前系统大多依赖运行中的启发式方法，例如按时间顺序修剪和工具输出掩码，或者在上下文即将达到限制时进行最终自我总结。启发式方法虽然成本低，但无法预见未来的依赖关系；总结虽然保留了叙述状态，但往往隐藏了确切的证据、定位器和可编辑的工件。我们提出了Self-GC，其中GC代表自治理上下文，同时刻意呼应垃圾回收概念：系统不仅回收未使用的令牌，还管理智能体上下文对象的生命周期。Self-GC将用户轮次、工具区间和技能状态转化为索引对象；要求侧通道规划器提出折叠、掩码和修剪操作；并由执行框架强制执行可恢复的侧车、安全的提交边界以及缓存感知的提交。在包含33个会话的Hard Set上，Self-GC修剪了43.95%的前缀令牌，同时使84.85%的未来延续不受影响，相比之下，启发式基线的无影响率为54.55%至69.70%。在包含332个会话的生产衍生套件上，三种规划器骨架的无影响率达到了91.27%至94.58%，而基线仍停留在77.71%至87.46%。在生产环境中，在线账户级拆分使日间平均输入令牌减少了10%至15%，峰值减少接近20%。这些结果表明，上下文管理应作为对索引化、可恢复对象的运行时生命周期控制，而非事后文本清理。

## Abstract
Long-horizon LLM agents accumulate tool results, files, plans, and user constraints that are too structured to be treated as a disposable text suffix. Current systems mostly rely on in-run heuristics such as chronological pruning and tool-output masking, or on final self-summary near a context limit. Heuristics are cheap but blind to future dependencies; summaries preserve narrative state but often hide exact evidence, locators, and editable artifacts. We present Self-GC, where GC denotes self-governing context while deliberately echoing garbage collection: the system does not merely reclaim unused tokens, but governs the lifecycle of agent context objects. Self-GC turns user turns, tool spans, and skill state into indexed objects; asks a side-channel planner to propose fold, mask, and prune actions; and lets the harness enforce recoverable sidecars, safe commit boundaries, and cache-aware commit. On a 33-session Hard Set, Self-GC prunes 43.95% of prefix tokens while leaving 84.85% of future continuations unaffected, compared with no-impact rates of 54.55% to 69.70% for heuristic baselines. On a 332-session production-derived suite, three planner backbones reach no-impact rates of 91.27% to 94.58%, while baselines remain at 77.71% to 87.46%. In production, an online account-level split reduces daytime average input tokens by 10% to 15%, with peak reductions near 20%. These results point to context management as runtime lifecycle control over indexed, recoverable objects rather than post hoc text cleanup.

## 1. 核心问题与整体含义

- **研究背景**：长时域LLM智能体（调用工具、编辑文件、多步工作流）会产生大量结构化上下文（工具结果、计划、用户约束）。当前主流上下文管理方法有两种：运行中启发式修剪（按时间顺序、类型等）和上下文接近限制时的最终自我总结。
- **根本问题**：启发式方法成本低但“盲目”于未来依赖关系；总结保留叙述但丢失了精确证据（URL、表格值、文件路径等）。两种方法均将历史视为线性令牌缓冲区，忽视了对象的生命周期差异。
- **整体含义**：Self-GC 提出将上下文视为**运行时对象的集合**，每个对象有不同的生命周期需求（可删除、可掩码、需精确恢复），从而将上下文管理从**事后文本清理**转变为**运行时生命周期控制**。

## 2. 方法论

- **核心思想**：将用户轮次、工具结果、技能状态映射为带稳定标识符的**索引对象**；通过**侧通道规划器**（side-channel planner）对对象提出折叠（fold）、掩码（mask）、修剪（prune）操作；**执行框架（harness）** 进行本地排练、强制可恢复侧车、维护安全提交边界和缓存感知提交。
- **关键技术细节**：
  - **索引对象**：`conversation:user:k`（用户请求+执行跨度）、`function:tool:n`（工具级跨度）。对象标识符由单调分配器生成，作为控制元数据。
  - **三种生命周期动作**：
    - **Fold**：将精确载荷移到侧车存储，留下紧凑恢复指针。
    - **Mask**：保留对象边界但省略重复或低信号中间内容。
    - **Prune**：移除过时内容，无恢复保证。
  - **规划-排练-提交流程**：
    1. 触发GC时，fork当前前缀并附加规划器指令。
    2. 规划器输出结构化计划（仅对现有标识符操作）。
    3. Harness本地排练：解析目标、删除无效或截断轮次操作、规范化重叠操作、生成投影、估计令牌节省。
    4. 仅在**安全轮次边界**提交：合并视图、修复父系祖先、持久化侧车、规范化provider消息。
  - **缓存感知提交**：使用公式 `CommitBenefit ≈ Nfuture(C - C′) - Lcache_break - LGC` 评估是否提交；当期望修剪超过0.3时立即提交，否则保持计划挂起。
  - **可恢复性**：折叠元数据作为控制平面提示附加到相关用户消息，而非助手散文，确保模型可见且不影响后续生成。

## 7. 优点

- **方法创新**：将上下文管理重新定义**为运行时对象生命周期控制**而非线性文本清理，概念新颖且符合实际需求。
- **安全设计**：规划-排练-提交分离，规划器仅提供语义判断，harness强制可恢复性、协议有效性和缓存感知，安全性高。
- **可恢复性**：Fold动作保留侧车存储的精确载荷，可被后续轮次引回，不同于总结丢失原始证据。
- **缓存经济性**：缓存感知提交策略减少不必要前缀缓存崩溃，平衡节省与开销。
- **实验充分**：同时提供了离线（有判官控制）和在线（真实流量）证据，且分析了失败模式。
- **框架可移植**：仅需少量上下文引擎钩子即可集成，不依赖具体provider。
