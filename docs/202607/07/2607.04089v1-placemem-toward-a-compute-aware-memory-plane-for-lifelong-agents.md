## 摘要
终身智能体需要的不仅仅是更大的上下文窗口和更好的检索。它们需要能够持久化、进化并在不强迫服务栈每一轮都重新计算相同历史或静默重用过时的运行时状态的情况下进行修正的记忆。我们将PLACEMEM作为终身智能体记忆的系统立场提出，并通过一个可执行的控制平面原型实例化。核心主张是，智能体记忆应表示为版本化的胶囊，在单一修正感知身份下统一语义、来源、有效性和可重用的运行时状态。当前原型中，胶囊驱动提示级文本检索、KV感知路由以及实时流式后端上的级联失效；前瞻性的层前沿重放被有意地构建为更深层次的集成议程，而非声称的引擎特性。我们描述了一个以vLLM为首的原型，具有持久化胶囊状态、并发安全失效、与OpenAI兼容的路由边车、类型化元数据契约以及一个用于测量实时首令牌延迟、重用率和修正后行为的基准测试框架。结果既是一个可执行工件，展示了当前修正感知的控制平面行为，也是一个具体路线图，用于未来终身智能体系统中重放感知的服务集成。

## Abstract
Lifelong agents need more than larger context windows and better retrieval. They need memories that can persist, evolve, and be corrected without forcing the serving stack to recompute the same history on every turn or silently reuse stale runtime state. We present PLACEMEM as a systems position on lifelong-agent memory, instantiated by an executable control-plane prototype. The central claim is that agent memory should be represented as versioned capsules that unify semantics, provenance, validity, and reusable runtime state under one correction-aware identity. In the current prototype, capsules drive prompt-level text retrieval, KV-aware routing, and cascading invalidation over live streamed backends; prospective layer-frontier replay is intentionally framed as a deeper integration agenda rather than a claimed engine feature. We describe a vLLM-first prototype with persistent capsule state, concurrency-safe invalidation, an OpenAI-compatible routing sidecar, a typed metadata contract, and a benchmark harness that measures live first-token latency, reuse, and post-correction behavior. The result is both an executable artifact that demonstrates correction-aware control-plane behavior today and a concrete roadmap for replay-aware serving integration in future lifelong-agent systems.

## 1. 论文的核心问题与整体含义（研究动机和背景）
- **核心问题**：终身智能体（lifelong agents）需要能持久化、进化并在修正后失效的记忆，但现有系统将**语义记忆**（事实、摘要等）和**运行时记忆**（前缀缓存、KV张量等）分离在两个独立层，导致两类可避免的失败：
  - 当存在语义相关且可重用的运行时状态时，仍从头重计算昂贵的预填（prefill）；
  - 在事实/摘要/计划被修正后，仍静默重用基于过期信息产生的运行时工件。
- **宏观含义**：遗忘不是便利性功能，而是保持行为正确性的基础设施。需要一个**计算感知的记忆平面**，使内存的语义和计算工件在单一身份下协同。
- **研究动机**：现有工作要么优化记忆语义（如Generative Agents、MemGPT），要么优化推理效率（如vLLM），但两者缺乏统一的控制平面。PLACEMEM填补了这一空白。

## 2. 论文提出的方法论

### 2.1 核心思想：版本化“记忆胶囊”
- 记忆胶囊（memory capsule）是一个**版本化对象**，它将语义内容（文本、摘要、来源、有效性窗口）与可重用的计算工件（嵌入、KV缓存段，未来可选层前沿检查点）绑定在**同一标识符**下。
- 一个胶囊驱动：
  - 提示级文本检索（语义相关）
  - KV感知路由（将请求导向持有相关KV工件的节点）
  - 级联失效（correction cascading invalidation）

### 2.2 关键技术细节
- **架构**（图1）：请求 → 时序检索器 → 胶囊目录 → 重放策略 → 放置边车 → vLLM后端；同时有热路径写回、胶囊更新、依赖图、失效引擎、基准信号。
- **效用函数**（公式1）：  
  \( U(c, t, r) = w_1P_{reuse} + w_2C_{saved} + w_3V_{sem} - w_4R_{stale} - w_5B_{remote} - w_6L_{target} \)  
  其中 \(P_{reuse}\) 估计近期重用概率，\(C_{saved}\) 是避免的计算量，\(V_{sem}\) 任务相关性，\(R_{stale}\) 新鲜度风险，\(B_{remote}\) 传输成本，\(L_{target}\) 延迟惩罚。控制平面根据该效用选择最优重放决策（文本检索/KV重放/未来层重放）。
- **失效机制**：维护胶囊间的依赖边（例如摘要依赖原始事实），当事实被修正时，级联失效传播到所有派生工件（摘要、索引、KV缓存），确保不再重用过期状态。

### 2.3 算法流程（文字说明）
1. 智能体发起新请求。
2. 时序检索器查询胶囊目录，获取相关胶囊候选。
3. 重放策略评估每个候选在目标节点上的各个重放模式（文本、KV、未来层）的效用值。
4. 选择最优决策，通过放置边车（placement sidecar）向vLLM后端发送元数据（胶囊ID、KV位置等）。
5. 后端根据元数据决定是否重用KV缓存或重新计算。
6. 当发生修正时，失效引擎读取依赖图，级联标记所有受影响胶囊及其派生工件为无效。
7. 后续请求将不会选择被无效的胶囊，从而实现正确性。

## 3. 实验设计

### 3.1 数据集 / 场景
- 三种**任务族**（均通过模拟生成48轮交互轨迹）：
  - **客服历史**：存储并修正退款政策。
  - **编程agent仓库循环**：跨轮次维护代码上下文和修正。
  - **研究agent事实修正**：逐步更新科学事实。
- 每个任务族包含一系列的“修正事件”，用于测试失效行为。

### 3.2 基准方法（baselines）
- **仅prompt执行**：每一轮从头计算，无任何记忆重用。
- **仅文本检索**：检索相关胶囊摘要并拼入提示，但不暴露KV工件。
- **仅运行时重用（无语义失效）**：基于vLLM前缀缓存重用KV，但不检查语义有效性和修正。
- **完整PLACEMEM**：胶囊驱动，带语义检索+KV路由+级联失效。

### 3.3 benchmark
- **Pilot评估**（表2）：使用模拟后端（固定SSE负载+可配置token间延迟），隔离控制平面开销。测量：首令牌延迟（TTFT）、KV重用率、修正后的陈旧命中次数、失效开销。
- **真实后端评估**（表3）：使用vLLM + LiquidAI/LFM2.5-230M模型。设置“修正追踪探测”：每个修正事件后发送9个查询（3个任务族×9=27轮后修正查询）。另外部署吞吐量验证：340轮批次，重复3次。
- 评估指标：平均TTFT、KV重用率、后修正准确率（模型输出是否包含修正后标识符）、陈旧选择次数（控制平面是否选择了过期胶囊）。

## 6. 论文的主要结论与发现
- **Pilot结果**（表2）：
  - 完整PLACEMEM使平均TTFT相比仅prompt降低60.7%（7.17 vs 18.25 ms），且KV重用率为1.0。
  - 仅运行时重用也大幅降低TTFT（6.72 ms），但产生17次陈旧重用；完整PLACEMEM消除所有陈旧重用，失效开销仅1.09 ms。
- **真实后端结果**（表3）：
  - 完整PLACEMEM达到与“重用无失效”相同的TTFT（20.54 vs 20.73 ms），且后修正准确率100%，而“重用无失效”准确率为0%（全部选择了陈旧摘要）。
  - 文本摘要基线准确率100%，但TTFT高75%（36.08 ms）。
- **主要发现**：将**记忆修正作为一阶操作**引入控制平面，可以在几乎不增加延迟的前提下完全避免重用过期状态，实现**修正感知的效率**。该论文证明概念有效，但重放引擎的深度集成是未来工作。

## 7. 优点
- **系统抽象创新**：首次提出“计算感知记忆平面”，将语义和运行时状态统一在版本化胶囊下，为终身智能体提供了新的设计思路。
- **可执行原型**：基于vLLM构建了完整控制平面（胶囊目录、依赖图、失效引擎、OpenAI兼容边车），可直接部署验证。
- **级联失效机制**：将记忆修正提升为第一类操作，支持派生工件的自动失效，纠正了现有系统“只更新文本不更新缓存”的缺陷。
- **效用函数显式建模**：将重放决策转化为多目标优化（语义价值、延迟、新鲜度、传输成本），优于纯字节匹配的缓存策略。
- **实验设计聚焦核心论题**：通过比较有无失效的版本，直接证明了修正感知控制平面的必要性，对比清晰。
- **坦诚说明当前局限**：明确区分“已实现的控制平面行为”和“未来引擎集成”，避免过度声明。

## 8. 不足与局限
- **实验规模小且模型极小**：仅使用230M参数模型，测试轨迹仅48轮+340轮批次，与真实终身智能体（数百轮、大型模型）差距大，结论的可推广性存疑。
- **模拟后端掩盖真实开销**：TTFT仅反映控制平面和边车路由时延，不包含GPU预填实际节省时间；真实后端的TTFT（~20 ms）也显著低于生产环境（可能预填需数秒），因此延迟收益可能被高估。
- **缺乏大规模异构成熟评估**：未在多节点、异构GPU集群上运行，效用函数中的远程传输成本 \(B_{remote}\) 和节点延迟 \(L_{target}\) 仅理论定义，未实验验证。
- **重放引擎未实现**：层前沿重放（layer-frontier replay）仅作为契约，未纳入实验，使得效果停留在提示级文本和KV路由层面，未来集成可能存在未预见的挑战。
- **未与现有语义缓存系统定量对比**：仅表1定性对比了GPTCache、SCALM，缺乏直接实验比较延迟、命中率、失效开销等。
- **隐私和安全性讨论仅一笔带过**：伦理声明中提到“持久记忆引发隐私问题”，但未在实验中评估任何隐私保护机制或滥用风险。
- **任务族覆盖有限**：仅三个模拟场景，未涉及多模态、工具使用、动态环境等更复杂的终身学习场景。
