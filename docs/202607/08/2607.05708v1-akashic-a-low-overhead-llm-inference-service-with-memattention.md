## 摘要
近期基于LLM的智能体系统在多轮交互、工具调用和跨会话工作流中不断累积上下文。为每个请求重放完整历史很快变得不切实际：长上下文增加了预填充成本，可能超出上下文限制，并且常常将任务相关证据埋没在不相关的内容中，降低了服务效率和输出质量。我们提出阿卡西，一种围绕MemAttention构建的低开销记忆系统，它将上下文组织为有界块，并建模跨块的语义关系，无需重复重写完整历史即可保留跨块证据。阿卡西进一步应用软硬件协同设计的内存布局，将可能共同检索的块放在一起，减少检索碎片化和I/O开销。在四个代表性工作负载和三种模型规模上，与强基线记忆系统相比，阿卡西将任务准确率提升高达10.2个百分点，吞吐量提升高达1.21倍，可持续请求率提升高达1.88倍。

## Abstract
Recent LLM-based agent systems continuously accumulate context across multi-turn interactions, tool invocations, and cross-session workflows. Replaying the full history for every request quickly becomes impractical: long contexts increase prefill cost, may exceed context limits, and often bury task-relevant evidence in irrelevant content, degrading both serving efficiency and output quality. We propose Akashic, a low-overhead memory system built around MemAttention, which organizes context into bounded chunks and models semantic relationships across chunks, preserving cross-chunk evidence without repeatedly rewriting the full history. Akashic further applies hardware-software co-designed memory placement to co-locate likely co-retrieved chunks, reducing retrieval fragmentation and I/O overhead. Across four representative workloads and three model sizes, Akashic improves task accuracy by up to 10.2 points, throughput by up to 1.21x, and sustainable request rate by up to 1.88x over strong prior memory baselines.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：LLM 智能体系统在多轮交互、工具调用和跨会话工作流中不断积累长上下文（long context）。为每个请求重放完整历史带来三大弊端：**预填充成本高**（prefill cost）、**可能超出上下文窗口**（context limit）、**无关内容掩埋关键证据**，导致服务效率与输出质量双下降。
- **背景与挑战**：现有记忆系统（如 Mem0、MemGAS、MemGPT、RMM）通过压缩或分段方式管理历史，但存在**粒度权衡**——全上下文压缩更新成本随历史增长，且混合无关主题引入噪声；分段压缩丢失跨段依赖，削弱多轮推理。此外，上下文信息密度非均匀（图2）：某些阶段高度可压缩（如 LoCoMo），另一些阶段信息密集（如 SWE-Bench），固定压缩策略不匹配实际负载。同时，语义相关的记忆在物理存储上可能分散在不同页面，造成**局部性差距**（locality gap），即使只检索少量记忆，若它们分散在多块，I/O 开销依然很高（图3、图4）。
- **整体含义**：需要一种低开销的记忆系统，同时优化语义构建与物理存储，在保持高质量推理的同时提升服务效率。Akashic 正是为此目标设计。

## 2. 论文提出的方法论

### 核心思想
- **MemAttention**：将上下文组织为**有界块**（bounded chunk），以块粒度增量维护记忆，避免重写全部历史；通过**跨块推理**（cross-chunk inference）保留跨块证据；采用**模型驱动的语义匹配**（model-driven relevance matching）同时用于维护与检索。
- **软硬件协同内存管理器**：根据检索共现模式，将可能共访问的块**物理共置**（co-locate），减少检索 I/O 和碎片化；采用**原地外更新**（out-of-place update）和后台**垃圾回收**（GC）回收空间。

### 关键技术细节
- **块粒度维护**：将输入上下文切分为固定大小（默认 1024 token）的块。活跃块累积新回合，达到阈值后触发压缩（compaction），仅处理该块（复杂度 O(L_chunk)），而非全部历史。
- **跨块推理**：压缩新块时，系统提取其元数据关键词，与同一用户-会话命名空间内已有块的元数据一同输入 LLM，由模型选出 top-5 最相关先验块，进行联合压缩（joint compaction），更新过时记忆或删除无效记忆。
- **模型驱动的语义匹配**：维护与检索使用相同的 relevance 函数——将查询/新块的元数据与候选块元数据送入 LLM，模型直接选择最相关的 top-5 块。免去手工设计相似度分数。
- **内存管理器算法**（Algorithm 2）：
  - **在线重定位**（online relocation）：联合推理时，若模型识别出一组应共置的块，则将它们异地写入新块，标记旧副本无效，更新逻辑目录。
  - **后台块压缩**（background block compaction）：当一块的无效比例 φ(B) ≥ 75% 时，提取所有有效块，按 (user_id, session_id) 分区，再调用 LLM 分组（LLMGroup）按亲和性重打包进新块，释放旧块。
- **维护与检索流程**（Algorithm 1）：新回合追加到活跃块 → 达到阈值时压缩、提取元数据、跨块查询 top-5 先验块、联合更新/删除 → 写回新块、清空活跃块 → 推理时：用当前上下文元数据查询所有候选块元数据，模型选择 top-5，获取块内容并按时间顺序拼接到当前上下文。

### 公式
- **保留比率**（Retained Ratio）：ρ = L_eff / L_raw，衡量原始上下文中有效信息的比例。
- **块无效比例**：φ(B) = N_invalid(B) / N_total(B)，用于触发垃圾回收。

## 3. 实验设计

- **数据集/场景**：四个代表性长时工作负载：
  - LoCoMo（长期多会话对话记忆，需时序与因果推理）
  - SWE-bench（高密度软件工程，解决真实 GitHub issue）
  - BrowseComp（突发性、交替密度模式的网页浏览轨迹）
  - WebArena（多步骤 web-agent 任务）
- **基准方法**（baselines）：
  - Full-context：无压缩或外部记忆，直接输入全部历史。
  - Mem0：全上下文记忆总结。
  - MemGAS：分段多粒度记忆关联与选择。
  - MemGPT：分层外部记忆，类似 OS 的虚拟上下文管理。
  - RMM：反思式记忆精炼（reflective memory refinement）。
- **评估指标**：**准确率**（各 benchmark 标准：LoCoMo 用 QA F1、SWE-bench 用 %Resolved、BrowseComp 用 Accuracy、WebArena 用任务成功率）和 **吞吐量**（token/s）。

## 6. 论文的主要结论与发现

- Akashic 在所有设置下均位于 **Pareto 前沿**，即同时实现更高准确率和更高吞吐量，或至少不劣于基线。
- **具体增益**（与最强基线相比）：
  - 任务准确率提升 **8.4–10.2 点**（LoCoMo）、**4.6–6.6 点**（SWE-bench）、**4.2–7.2 点**（BrowseComp）。
  - 单请求吞吐量提升 **1.21×–1.35×**。
  - 并发场景下可持续请求率提升 **1.26×–1.74×**（最高 1.88× 对 Mem0）。
- **消融证实**：块级维护是效率主要来源，跨块联合提升质量且开销可接受，模型驱动匹配优于纯稠密嵌入。
- **内存管理器**有效缩小局部性差距：p95 冷缓存检索延迟降低 17%–42%，同时空间/写放大仅 1.17×/1.34×，优于激进重定位策略。

## 7. 优点

- **方法创新性**：首次将有界块增量维护与跨块语义推理结合，并统一用 LLM 自身进行相关性选择，无需额外嵌入模型或手工特征。
- **系统设计优雅**：将内存管理的逻辑层面（语义构建）与物理层面（存储布局）分离，通过模型驱动的重定位和后台压缩优化局部性，类似 LSM-tree 的 compaction 思想。
- **实验全面、结果显著**：在多种负载、模型和并发条件下验证，增益量化清晰（准确率、吞吐、延迟），消融和敏感性分析充分，超参数鲁棒。
- **开源承诺**：计划将代码合并到 RedKnot 框架，便于复现与扩展。

## 8. 不足与局限

- **模型覆盖有限**：仅测试了 Qwen-8B、OPT-30B、Llama-70B 三种，未涵盖更大规模（如 130B+）或最近前沿模型（如 GPT-4、Claude 系列），性能趋势可能随模型规模变化。
- **检索依赖 LLM 本身**：模型驱动的语义匹配需要额外 LLM 调用（每次维护和检索各一次 top-5 选择），虽然开销小（仅元数据输入），但仍增加 latentcy，且 LLM 的选择质量未与专用嵌入模型对比。
- **阈值依靠启发式**：块大小（1024）、top-p（5）、GC 无效比（0.75）等基于经验设定，未在更广泛场景验证其普适性。
- **未评估训练微调**：论文关注推理服务，未涉及对记忆模块的微调或联合训练，可能限制了在特定任务上的记忆质量。
- **高密度场景优势减弱**：在 SWE-bench 上，吞吐量提升相对较小，且准确率增幅也较 LoCoMo 低，说明 Akashic 更擅长处理可压缩、非均匀密度的场景。
- **跨会话内存共享**：实验仅考虑单个用户-会话命名空间，未探讨跨用户或跨会话的记忆迁移与共享优化。
