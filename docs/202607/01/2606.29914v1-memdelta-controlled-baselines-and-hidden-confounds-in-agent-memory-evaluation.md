## 摘要
代理记忆系统越来越多地针对RAG和全上下文基线进行评估，但报告的性能提升往往混合了记忆方法的改变与语言模型、嵌入模型或检索管道的改变，使得实际衡量的内容变得不明确。我们提出MemDelta，一种受控评估协议，在LongMemEval-S（500个问题，50+会话，三个模型家族）上每次只变化一个组件。四个发现如下：(1) 逐字RAG匹配全上下文GPT-4o-mini（47.2% vs. 49.8%，p = 0.34），但排名在不同模型间反转：Gemini从全上下文中获得+14pp的提升，而Sonnet从RAG中获得+31pp的提升，部分原因是它拒绝了63%的全上下文查询；(2) 在相同管道中仅替换嵌入模型，在n=500时准确率变化+6.2pp（p = 0.004），Mem0优于MiniLM-RAG达+11pp，但落后于cloud-RAG 1.2pp，因此一个变量即可翻转结论；(3) 代理自记忆（42%）表现不及基本检索（47%）；(4) 在6种问题类型中的2种（n=88）上，Mem0以50倍成本匹配cloud RAG（72.7% vs. 73.9%，p = 1.0），表明其增益狭窄而非普遍。我们建议记忆评估在比较中固定嵌入模型，按模型家族分层，并在将增益归因于架构前报告写入路径成本。

## Abstract
Agent memory systems are increasingly evaluated against RAG and full-context baselines, but reported gains often mix changes in the memory method with changes in the language model, embedding model, or retrieval pipeline, making it unclear what is actually being measured. We present MemDelta, a controlled evaluation protocol that varies one component at a time on LongMemEval-S (500 questions, 50+ sessions, three model families). Four findings emerge: (1) verbatim RAG matches full-context GPT-4o-mini (47.2% vs. 49.8%, p = 0.34), but the ranking reverses across models: Gemini gains +14pp from full context, while Sonnet gains +31pp from RAG, partly because it refuses 63% of full-context queries; (2) swapping only the embedding model in an identical pipeline shifts accuracy by +6.2pp at n = 500 (p = 0.004), and Mem0 beats MiniLM-RAG by +11pp but loses to cloud-RAG by 1.2pp, so one variable flips the conclusion; (3) agent self-memory (42%) underperforms basic retrieval (47%); (4) on 2 of 6 question types (n = 88), Mem0 matches cloud RAG (72.7% vs. 73.9%, p = 1.0) at 50x the cost, suggesting narrow rather than general gains. We recommend memory evaluations fix embedding models across comparisons, stratify by model family, and report write-path cost before attributing gains to architecture.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究动机**：当前代理记忆系统（如 Mem0）的评估往往同时改变记忆方法、语言模型、嵌入模型或检索管道等多个组件，导致报告的性能提升无法归因到具体哪个组件。例如，Mem0 在 MiniLM 嵌入下比 RAG 高 11pp，但换成云嵌入（text-embedding-3-small）后反而低 1.2pp——结论完全翻转。这种“隐藏的混杂因素”使得评估结果不可靠、不可重复。
- **整体含义**：需要一套受控评估协议，每次只改变一个变量，以便区分架构增益和工程优化（如嵌入质量、模型行为、计算成本）的贡献。论文旨在揭示现有评估中常见的四个混杂因素（检索质量、嵌入选择、模型行为、写路径成本），并提供相应的控制方法。

## 6. 论文的主要结论与发现

1. **检索质量**：从无记忆（2.2%）到基本 RAG（47.2%）跃升 +45pp，占绝对效果的大部分。逐字 RAG 在 GPT-4o-mini 上匹配全上下文（49.8%），差距不显著（p=0.34）。
2. **模型行为**：全上下文 vs RAG 的排名完全依模型而异：Gemini 上全上下文 +14pp，Sonnet 上 RAG +31pp（因 63% 全上下文查询被拒绝）。同一结论在不同模型上反转。

4. **代理自记忆**（S2，42%）弱于基本检索（S4，47%），划痕板压缩失败，特别是多会话问题仅 3.3%。
5. **写路径成本**：在可匹配的 88 个实例上，Mem0（72.7%）与 cloud RAG（73.9%）无显著差异（p=1.0），但成本高 50 倍（$0.50+ vs $0.01）。Mem0 的优势仅出现在 SS-User 和多会话问题上，并非普遍提升。
6. **每类型分析**：无单一策略占优。检索擅长单会话问题（84–87%），全上下文在知识更新（72%）和多会话（37%）上有优势，但 cloud 嵌入缩小了差距。

## 8. 不足与局限

- **实验覆盖不足**：
  - Mem0 仅评估了 2/6 的问题类型（SS-User 和 Multi-Session，n=88），时间推理和知识更新未覆盖，这些类型可能是提取管道的自然优势场景。

- **缺乏因果分解模型**：论文明确指出比较是“受控探针”而非加法因果模型，变量间可能存在交互未被建模。
- **实际部署假设**：忽略写路径成本可能在不频繁检索的场景下被稀释，但论文仍强调成本在任何部署中都会累积。
