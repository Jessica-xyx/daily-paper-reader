## 摘要
LLM智能体基准测试衡量任务完成度、可靠性和推理成本，但未考虑智能体运行在磁盘上留下的持久化数据，包括日志、上下文快照、检查点和调试跟踪。我们提出了AgentFootprint，一个跨框架的运行后智能体存储占用基准测试。其序列化感知指标套件测量总保留量、通道组成、重复性、增长量、可压缩性和对话历史可重建性。它解决了一个测量陷阱：由于数据库分页和JSON转义掩盖了重复内容，原始的字节级测量将重复性低估了一个数量级。一个固定轨迹控制将智能体生成的逻辑卷与持久化层的放大效应分离：通过七个持久化框架重放同一轨迹，产生了6.7倍的差异。在相同模型、工具和任务下，准确率100%的配置在保留字节上差异达15.7倍，尽管它们的默认设置支持不同的恢复和审计能力。三个完整历史配置在重复观察压力任务上呈超线性增长。来自108个实例归一化的SWE-bench Verified提交的导出轨迹，每个实例跨越三个数量级，与解决率无显著相关性。一个内容寻址存储将保留量减少4.8倍至32.7倍，同时保持每个可重建性得分。这些结果将持久化存储确立为一个资源指标，应与准确率和可重建性一同报告。

## Abstract
LLM agent benchmarks measure task completion, reliability, and inference cost, but not the persistent data an agent run leaves on disk, including logs, context snapshots, checkpoints, and debug traces. We introduce AgentFootprint, a cross-framework benchmark of post-run agent storage footprint. Its serialization-aware metric suite measures total retention, channel composition, duplication, growth, compressibility, and conversation-history reconstructability. It addresses a measurement trap: naive byte-level measurement understates duplication by an order of magnitude because database paging and JSON escaping obscure repeated content. A fixed-trace control separates agent-generated logical volume from persistence-layer amplification: replaying the same trajectory through seven persisting frameworks yields a 6.7x spread. Under identical models, tools, and tasks, configurations with 100% accuracy differ by 15.7x in retained bytes, although their defaults support different recovery and audit capabilities. Three full-history configurations grow superlinearly on a repeated-observation stress task. Exported trajectories from 108 instance-normalized SWE-bench Verified submissions span three orders of magnitude per instance, with no detectable correlation with resolve rate. A content-addressed store reduces retention by 4.8x-32.7x while preserving every reconstructability score. These results establish persistent storage as a resource metric to report jointly with accuracy and reconstructability.

### 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：当前 LLM 智能体评估仅关注任务完成率、可靠性和推理成本，完全忽略了智能体执行后留在磁盘上的持久化数据（日志、上下文快照、检查点、调试跟踪等）。这些“隐藏足迹”直接影响桌面部署可行性、数据保留合规性以及大规模集群的运维成本。
- **研究动机**：存储占用如同移动应用的内存占用一样，初期无人关注，但当部署规模扩大时就会成为瓶颈。作者通过测量一个生产环境中的业务系统发现，单任务保留 131 MB 而交付工件仅 2.6 MB；一个约 1000 轮的部署已累积约 56 GB。若以每天 10,000 轮计算，默认配置下每天产生 51 GB 存储，而仅保留历史的最低配置只需 3.1 GB，差异达 16 倍。
- **整体含义**：持久化存储应成为与准确率、推理成本并列的 LLM 智能体评估指标，且需要一套可重复、可比较的度量方法。

### 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：设计一个跨框架的运行后存储占用基准，提出序列化感知的指标套件，并解决三个挑战：**归属**（哪些字节属于哪个任务）、**序列化掩蔽**（SQLite 页面、JSON 转义掩盖了重复内容）、**语义**（字节数本身不能区分“有益冗余”与“浪费”）。
- **关键技术与指标**（六个指标）：
  - **总保留量 (Stotal)**：工作区与框架侧残留的字节总和。
  - **存储通道组成**：按确定性路径规则将 Stotal 分组为状态/检查点、调试/事件日志、对话/动作等。
  - **重复因子 (D)**：逻辑流字节总数除以唯一内容块字节总数。使用 FastCDC 分块 + SHA-256，在提取的逻辑流（SQLite 单元、JSONL 行等）上计算，避免原始文件碎片导致的低估。**关键发现**：原始字节级 chunking 会将 D 低估为 1.01（实际逻辑流下 D=12.1），因为 SQLite 页面和 JSON 转义掩蔽了重复。
  - **增长指数 (α)**：在重复观察压力任务（T=25,50,100,200）上拟合 log Stotal ~ α log T，描述存储如何随轮次增长。
  - **可压缩性 (C)**：所有保留文件拼接后用 zstd-19（128MB 窗口）压缩的比率，提供语法层面的“免费午餐”上限。
  - **对话历史可重建性 (R)**：从 0 到 3 的评分，判断保留的字节能否重建模型在每步接收到的对话历史。R=3 表示至少一个通道能完全精确恢复；通过代理日志记录真实请求体，逐字段比对验证。
- **固定轨迹控制**：使用脚本化的模拟端点回放完全相同的轨迹（三个读操作、182.7 KB 观察、一个答案），通过 7 个持久化框架的未修改适配器，隔离持久层的放大效应。

### 3. 实验设计：数据集/场景、Benchmark、对比方法

- **任务套件（Benchmark）**：AgentFootprint 包含六族任务加上一个控制：
  - **file-QA 套件**（10 个任务）：约 600 KB 文档库，5 个顺序问题，要求重读文件（人为放大读取次数）。
  - **long-horizon 套件**：重复读取一个 2 KB 状态文件，轮次 T=25,50,100,200，测量增长指数 α。
  - **write-task 套件**（5 个任务）：读两个文档并输出一个摘要文件，测量残差（框架残留）与交付工件的比值。
  - **exploratory-retrieval 套件**（5 个任务）：去除对抗性元素，自由探索。
  - **heterogeneous 套件**（5 个编辑任务 + 5 个数据分析任务）：读-变换-写，联合评判正确性和工件交付。
  - **ablation 运行**：禁用持久化，重新执行 file-QA 套件。
- **对比的框架**（8 个）：LangGraph、AutoGen、CrewAI、SmolAgents、OpenAI Agents SDK、LlamaIndex、Agno、InfiAgent。均使用文档中描述的“持久会话配置”（即默认持久化配置），相同的后端模型（DeepSeek-V4-Flash，温度 0）、相同的工具集合（list, read, write）、相同任务文本，每个条件 3 次重复。
- **野外数据**：SWE-bench Verified 的 134 个提交（108 个实例归一化），分析每个系统的导出轨迹体积与解决率的关系。

### 4. 资源与算力

- **实验数量**：主研究 509 次沙盒运行，加上 552 次复制和变体（不同后端、增长种子、历史保留形式、共享线程、回放来源等），总计 **1061 次沙盒运行**。另加 SWE-bench 的 134 个提交分析。
- **充分性分析**：

### 6. 论文的主要结论与发现

1. **相等准确率下的大分散**：在 file-QA 套件上，所有达到 100% 准确率的框架，保留字节差异达 **15.7 倍**（LangGraph vs. CrewAI）。即使仅考虑历史可重建性 (R=3)，也达 3.9 倍。
2. **固定轨迹下持久层放大差异**：相同轨迹通过 7 个持久化框架，保留字节差异达 **6.7 倍**，证明持久层设计本身是主要贡献者。
3. **增长模式**：完整历史持久化框架（LangGraph、AutoGen、LlamaIndex）在重复观察任务上呈超线性增长（α=1.74-1.95），而窗口化或追加型设计保持近线性（α=0.73-1.20）。200 轮后 LangGraph 保留 323 MB，而 OpenAI Agents 仅保留 1.4 MB。
4. **残差支配交付工件**：在 write-task 套件上，框架残差是交付工件的 **144 倍到 24,033 倍**。生产系统测量中残差约为交付工件的 30 倍。
5. **野外数据无存储-成功相关性**：108 个 SWE-bench 实例归一化系统中，每实例导出轨迹体积跨度 1,617 倍，与解决率的 Kendall τ=0.080（p=0.222），无显著相关。
6. **内容寻址存储可大幅减少**：一个约 300 行的后处理内容寻址压缩器，在保持所有可重建性评分不变的前提下，将保留量减少 **4.8 倍至 32.7 倍**。

### 7. 优点

- **首创性**：第一个系统性的、跨框架的 LLM 智能体运行后存储占用基准测试。
- **方法论严谨**：序列化感知的度量解决了原始字节测量低估重复性的问题；通过逻辑流提取和内容探针纠正了 SQLite 和 JSON 的掩蔽效应。
- **分离逻辑体积与持久层放大**：固定轨迹控制实现了因果归因，证明了 6.7 倍的纯粹持久层差异。
- **兼顾受控与野外**：既在受控实验室环境中进行多维度测量，又在真实世界（SWE-bench）中验证了差异。
- **重建性验证**：通过代理日志逐字段比对和进程边界恢复实验，建立了 R 评分的实际意义。
- **公开可复现**：所有代码、适配器、数据生成器和分析脚本作为附属材料发布，支持独立验证。

### 8. 不足与局限

- **任务覆盖有限**：受控套件只覆盖检索和保留任务，未涵盖多步推理、代码生成、工具调用错误恢复等复杂场景。
- **框架版本依赖**：结论基于特定版本的框架和其默认持久化配置。未来版本可能改变行为。
- **配置敏感性**：作者承认 AutoGen 的保存频率选项是“最具争议的”选择，不同保存策略可能改变结果水平。此外，禁用了 OpenAI Agents 的托管跟踪（远程保留，不在本地边界内），但未比较其效果。
- **野外数据不足**：SWE-bench 数据是导出轨迹而非运行时全部残留，且不同系统的导出策略不同，不能直接比较运行时足迹。
- **R 评分范围有限**：R 仅衡量对话历史可重建性，不包含工作流状态恢复、工具状态恢复或调试来源功能。
- **共享线程成本被低估**：沙盒模式忽略了长期共享进程的累积成本（如 LangGraph 在共享线程中 10 个任务保留 95.2 MB，而沙盒模式仅 5.8 MB）。
- **数据保留与隐私问题**：内容寻址存储可能引入跨租户的隐私泄露和删除复杂性，文中只简要提及但未深入讨论。
- **计算资源方面**：未报告 API 调用成本或其他间接开销，仅聚焦于磁盘存储。
