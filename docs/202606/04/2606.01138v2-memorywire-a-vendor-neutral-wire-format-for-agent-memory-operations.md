## 摘要
智能体记忆框架——mem0、Letta/MemGPT、Cognee、Zep/Graphiti、MemoryOS、MemTensor——各自提供自己的SDK、存储布局和操作词汇。目前没有共享的消息格式：每次集成都是定制的，每次迁移都从头重建记忆，并且没有框架提供治理界面让人类在写入进入长期存储之前进行审查。我们提出memorywire，这是一种基于JSON-Schema 2020-12的消息格式，支持四种记忆类型（语义、情景、程序、情感）上的五种记忆操作（记忆、回忆、遗忘、合并、过期），并包含MemoryStore接口、扇出路由器和可选的HITL治理通道。我们描述了一个开源参考实现，包含五个后端适配器（sqlite-vec、mem0、Letta、Cognee、pgvector）；在一个包含100个事实/50个查询的标注语料库（其中42个具有非空黄金ID和8个无匹配探针）上的微基准测试中，对42个黄金ID查询实现了recall@5=1.000，摄取p50=37.8毫秒，回忆p50=40.6毫秒；一个对抗性融合实验表明，在1-of-N排名0注入扫描（K∈{0,5,...,50}）中，倒数排名融合在recall@5=1.000上保持稳定，而最大融合在K≥5时崩溃至0.500且泄漏率为80%；以及一个包含16个场景的跨适配器一致性测试套件，在80个测试单元中通过68个，零失败。这项贡献并非新算法，而是将现有组件（RRF、FSM、STM/LTM整合、批准-修改工作流）封装成一个与供应商无关的协议，并通过经验验证的参考实现，旨在与模型上下文协议协同而非竞争。

## Abstract
Agent-memory frameworks -- mem0, Letta/MemGPT, Cognee, Zep/Graphiti, MemoryOS, MemTensor -- each ship their own SDK, storage layout, and operational vocabulary. There is no shared wire format: every integration is bespoke, every migration rebuilds memory from scratch, and no framework ships a governance surface that lets a human review writes before they enter long-term storage. We present memorywire, a JSON-Schema 2020-12 wire format for five memory operations (remember, recall, forget, merge, expire) over four memory types (semantic, episodic, procedural, emotional), with a MemoryStore interface, a fan-out router, and an optional HITL governance channel. We describe an open-source reference implementation with five backend adapters (sqlite-vec, mem0, Letta, Cognee, pgvector); a microbenchmark on a 100-fact / 50-query labelled corpus (42 with non-empty gold ids + 8 no-match probes) achieving recall@5 = 1.000 on the 42 gold-id queries with ingest p50 = 37.8 ms and recall p50 = 40.6 ms; an adversarial-fusion experiment showing Reciprocal Rank Fusion holds recall@5 = 1.000 across a 1-of-N rank-0 injection sweep (K in {0, 5, ..., 50}) where max fusion collapses to 0.500 with 80% leak at K >= 5; and a 16-scenario cross-adapter conformance suite passing 68 of 80 cells with zero failures. The contribution is not a new algorithm; it is a packaging of established components (RRF, FSMs, STM/LTM consolidation, diff-and-approve workflows) into a venue-neutral protocol with an empirically validated reference, positioned to compose with the Model Context Protocol rather than compete with it.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**：现有智能体记忆框架（如 mem0、Letta/MemGPT、Cognee、Zep/Graphiti、MemoryOS、MemTensor）各自定义独立的 SDK、存储布局和操作词汇，缺乏共享的线格式（wire format）。导致每次集成都是定制开发，迁移记忆需从零重建，且没有框架提供人工审核写入的治理界面。
- **背景**：MCP（模型上下文协议）为标准化的工具调用提供了成功范例，但未定义记忆原语。记忆操作（写入、召回、遗忘、合并、过期）需要比工具调用更丰富的生命周期和类型体系，因此需要一个独立的、与供应商无关的线格式。
- **意义**：通过标准化记忆操作和类型，降低框架间集成成本，支持跨运行时携带记忆，并首次在协议层引入可选的“差异-审核-批准”治理通道，提升安全性和可审计性。

## 2. 提出的方法论：核心思想、关键技术细节

- **核心思想**：将记忆操作抽象为五种操作（remember、recall、forget、merge、expire）和四种记忆类型（semantic、episodic、procedural、emotional），用 JSON Schema 2020-12 定义请求/响应格式。提供 MemoryStore 协议接口和扇出路由器（fan-out router），后端只需实现该协议即可接入。
- **关键技术细节**：
  - **操作类型**：remember（写入）、recall（读取）、forget（删除，支持软删除/硬删除）、merge（去重合并）、expire（TTL 策略执行）。
  - **记忆类型**：语义（事实）、情景（时间/地点背景）、程序（FSM 编码的步骤）、情感（情感关联）。
  - **扇出路由器**：并行分发操作到多个后端。recall 结果融合默认使用“倒数排名融合”（RRF，公式 `score(d)=∑₁/(k+rᵢ(d))`，k=60），同时提供 MAX 和 WEIGHTED 融合选项，以及可选的单跳图增强。
  - **部分失败模型**：使用 `asyncio.gather(return_exceptions=True)`，单个后端超时或报错不影响整体操作。
  - **可选治理通道**：当 `approval_required=true` 时，请求先被暂存（sentinel 值标记为待审核），通过 UI 展示差异（diff），经人工批准或拒绝后才写入。
  - **参考实现**：Python 3.11+，适配五个后端（sqlite-vec、mem0、Letta、Cognee、pgvector），提供 STM→LTM 后台转换器、程序记忆 FSM 后端、星舰治理界面。

- **对抗性融合实验（Adversarial Fusion）**：

  - 参数：K 从 0 到 50 扫描，20 个查询（每查询 2 个黄金 ID），种子固定为 1337。

## 6. 主要结论与发现

- **微基准测试**：recall@5 = 1.000（42 个有黄金 ID 的查询），写入 p50=37.8ms，召回 p50=40.6ms，均低于设定的 SLA 目标。8 个无匹配探针均返回非空——因为 v0 没有设定相关性阈值，设计上故意返回 top-k。
- **对抗融合**：RRF 在整个 K 扫描中保持 recall@5=1.000，无攻击泄漏；MAX 在 K≥5 时 recall@5 降至 0.500，泄漏率 80%。WEIGHTED（等权重）表现同 RRF。结论：RRF 因忽略原始分数、仅依赖排名，天然抵抗 1-of-N rank-0 注入攻击（多数良性后端同意下）。
- **一致性测试**：68/80 通过，0 失败，证明协议对多个不同后端（SQL-backed vs SDK-wrapped）的描述力足够。跳过项暴露了规范中需强化的点（空策略保护、稳定 ID 必现、用户命名空间）。
- **协议定位**：memorywire 不是算法创新，而是将 RRF、FSM、STM/LTM 整合、差异-审批工作流等成熟组件封装为中立协议，可与 MCP 组合（作为 MCP 工具、资源或未来扩展），而非竞争。

## 7. 优点

- **标准化价值**：首次为智能体记忆操作提供供应商中立的线格式，降低了框架间集成和迁移的成本。
- **完整配套**：包括规范、参考实现（5 个后端）、威胁模型、一致性测试、开放数据，共同构成一个可复制、可扩展的生态。
- **安全性设计**：内置可选治理通道（差异-审批）、RRF 的抗注入鲁棒性、程序记忆 FSM 的沙箱化（限制回调键）、审计日志只追加等，考虑了实际部署中的常见威胁。
- **实验多样**：虽然规模有限，但覆盖了微基准、对抗安全、适配器兼容性、初步记忆基准，为后续工作提供了可复现基线。
- **清晰的前置声明**：作者明确限定本工作不是新算法、不是 LLM 贡献、不是稳定版本，允许 v0 至 v0.5 破坏性变更，显示了学术诚信。

## 8. 不足与局限

- **评估规模小**：微基准语料仅 100 事实，对抗实验仅 3 个后端和 1 个种子，LongMemEval/LoCoMo 仅为初步（1 种子，少量问题），不足以支撑统计显著性结论。
- **评估环境单一**：所有实验在单台笔记本 CPU 上完成，未测试分布式、网络延迟、大规模部署场景。
- **缺乏外部审计**：作者自行扮演评估者，用户研究（如治理 UI 可用性）尚未进行。
- **基准选择局限**：未使用 BEAM（因找不到规范数据集），LongMemEval 和 LoCoMo 非完整版，嵌入器是弱小型（all-MiniLM-L6-v2），结果代表该嵌入器的上限，而非 memorywire 协议的上限。
- **一致性测试中的跳过项**：暴露了 v0 规范的部分模糊性（如空策略需抛出异常、后端必须返回稳定 ID、用户命名空间可选），需要在 v0.2 中修订。
- **对抗模型假设较强**：RRF 抗注入攻击依赖于“多数后端是良性的且对黄金 ID 达成共识”；如果恶意后端数量≥良性，或攻击者能伪造出现在良性结果集中的 ID，RRF 的鲁棒性会减弱。文中未评估这些更复杂的攻击场景。
- **协议成熟度**：目前 v0，允许破坏性变更，且尚未被任何主要框架采用；将其作为 MCP 扩展或 IETF 草案的路径仍为意向，尚无正式承诺。
