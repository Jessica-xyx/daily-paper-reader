## 摘要
智能体内存框架——mem0、Letta/MemGPT、Cognee、Zep/Graphiti、MemoryOS、MemTensor——各自提供自己的SDK、存储布局和操作词汇。目前没有共享的线格式：每次集成都是定制的，每次迁移都从头重建内存，也没有框架提供治理界面，让人类在写入进入长期存储之前进行审查。我们提出memorywire，一种基于JSON-Schema 2020-12的线格式，用于五类内存操作（记住、回忆、遗忘、合并、过期），涵盖四种内存类型（语义、情景、程序、情感），并提供MemoryStore接口、扇出路由器和可选的HITL治理通道。我们描述了一个开源参考实现，包含五个后端适配器（sqlite-vec、mem0、Letta、Cognee、pgvector）；在包含100个事实和50个查询的标注语料库上的微基准测试中，对42个标注查询实现了recall@5=1.000，摄入p50=37.8毫秒，回忆p50=40.6毫秒；一项对抗融合实验表明，在1-of-N rank-0注入扫描（K∈{0,5,...,50}）中，倒数排序融合（RRF）保持recall@5=1.000，而最大融合在K≥5时降至0.500并出现80%泄漏；以及一个包含16个场景的跨适配器一致性测试套件，通过了80个单元中的68个，零失败。本贡献并非新算法，而是将已有组件（RRF、有限状态机、短期/长期记忆整合、差异审批工作流）打包成一种场合中立的协议，并附有经验验证的参考实现，旨在与模型上下文协议（MCP）互补而非竞争。

## Abstract
Agent-memory frameworks - mem0, Letta/MemGPT, Cognee, Zep/Graphiti, MemoryOS, MemTensor - each ship their own SDK, storage layout, and operational vocabulary. There is no shared wire format: every integration is bespoke, every migration rebuilds memory from scratch, and no framework ships a governance surface that lets a human review writes before they enter long-term storage. We present memorywire, a JSON-Schema 2020-12 wire format for five memory operations (remember, recall, forget, merge, expire) over four memory types (semantic, episodic, procedural, emotional), with a MemoryStore interface, a fan-out router, and an optional HITL governance channel. We describe an open-source reference implementation with five backend adapters (sqlite-vec, mem0, Letta, Cognee, pgvector); a microbenchmark on a 100-fact / 50-query labelled corpus achieving recall@5 = 1.000 on the 42 labelled queries with ingest p50 = 37.8 ms and recall p50 = 40.6 ms; an adversarial-fusion experiment showing Reciprocal Rank Fusion holds recall@5 = 1.000 across a 1-of-N rank-0 injection sweep (K in {0,5,...,50}) where max fusion collapses to 0.500 with 80% leak at K >= 5; and a 16-scenario cross-adapter conformance suite passing 68 of 80 cells with zero failures. The contribution is not a new algorithm; it is a packaging of established components (RRF, FSMs, STM/LTM consolidation, diff-and-approve workflows) into a venue-neutral protocol with an empirically validated reference, positioned to compose with the Model Context Protocol rather than compete with it.

### 1. 核心问题与整体含义（研究动机和背景）

*   **核心问题**：当前智能体（Agent）的长期记忆框架（如 mem0, Letta, Cognee 等）存在严重的“孤岛”现象。每个框架都定义了各自私有的 SDK、存储结构、操作术语和数据模型。
*   **导致后果**：
    *   **集成成本高**：将不同的记忆框架集成到同一个智能体系统中需要定制化开发。
    *   **迁移困难**：将智能体从一个框架迁移到另一个框架，需要重写数据迁移工具，且可能丢失源框架特有的一些状态。
    *   **缺乏治理**：没有一个框架提供“人在回路中（HITL）”的治理界面，让人类在记忆写入长期存储前进行审查和批准。
*   **整体含义**：为了解决这些问题，论文提出了 **memorywire**，一个供应商中立的、标准化的**线格式**（Wire Format）。它的目标不是发明新的算法，而是将已有的成熟组件（如RRF融合、有限状态机、短期/长期记忆整合）打包成一个统一的、可互操作的协议，旨在与模型上下文协议（MCP）互补，而非竞争。

### 2. 方法论：核心思想、关键技术细节

*   **核心思想**：定义一套通用的、与具体实现无关的协议，用于智能体的内存操作。所有参与方都遵循此协议进行通信。
*   **技术细节**：
    *   **线格式**：基于 **JSON Schema 2020-12** 定义了五种内存操作和四种内存类型的请求/响应格式。
    *   **五种操作（5 Memories Operations）**：
        1.  `remember`（记住）：写入一条新记忆。
        2.  `recall`（回忆）：根据查询检索记忆。
        3.  `forget`（遗忘）：删除一条或多条记忆（默认软删除，支持硬删除）。
        4.  `merge`（合并）：合并重复的记忆。
        5.  `expire`（过期）：根据TTL策略自动清理旧记忆。
    *   **四种记忆类型（4 Memories Types）**：
        1.  `semantic`（语义）：陈述性事实，如“Alice对花生过敏”。
        2.  `episodic`（情景）：具有时间/地点的事件，如“2026年5月20日，Alice告诉我她对飞行感到紧张”。
        3.  `procedural`（程序）：如何做一件事的FSM编码流程。
        4.  `emotional`（情感）：情感关联，如“Alice在讨论飞行时表现出焦虑”。
    *   **MemoryStore协议**：一个Python接口，所有后端适配器都必须实现，包括 `remember`, `recall`, `forget`, `merge`, `expire`, `health` 和 `capabilities`（声明支持的功能，如vector, FTS等）。
    *   **Memory Router（内存路由器）**：一个核心组件，本身也是一个MemoryStore。它可以将一个操作扇出（fan-out）到多个子存储后端，并对`recall`的结果进行融合。
    *   **融合算法（Fusion Math）**：
        *   **RRF (默认)**：使用倒数排序融合（Reciprocal Rank Fusion），公式为 `score(d) = sum_{i in S} 1 / (k + r_i(d))`，其中 `k=60`。优点是抗恶意注入能力强，因为只关注排名，不关注原始分数。
        *   **MAX**：取各存储后端返回的最高分。对恶意注入敏感。
        *   **WEIGHTED**：对各存储后端的结果进行加权求和。对恶意注入敏感。
    *   **治理通道（Governance Channel）**：一个可选组件。当 `approval_required=True` 时，`remember`, `forget`, `merge` 操作会被暂存到一个待审批（PENDING_APPROVAL）队列，`recall` 操作无法看到这些暂存记录。必须通过治理UI上的审查员（Reviewer）批准后，这些写操作才会生效。

### 3. 实验设计

### 6. 主要结论与发现

1.  **协议可行性**：提出的memorywire规范从技术上是可行的，其参考实现能够在多种后端上正常工作，性能开销在可接受范围内（微基准测试中recall p50 = 40.6ms）。
2.  **RRF的抗攻击性**：在“1-of-N”恶意后端注入攻击下，RRF融合算法表现出极强的稳健性，在K攻击预算从0到50的扫描中，recall@5始终保持1.000。而MAX算法在K>=5时就立刻崩溃，recall@5降低至0.500，80%的返回结果为攻击者控制。
3.  **SQL后端兼容性最好**：完全自主控制数据库schema的后端（sqlite-vec, pgvector）能完美支持memorywire协议的所有特性。而封装现有SDK的后端（mem0, Letta, Cognee）在某些场景下（如不支持`user_id`过滤、不支持空策略校验、不支持稳定记录ID）必须跳过测试。
4.  **规范可改进性**：跨适配器合规测试成功暴露了v0版规范的漏洞，例如：
    *   要求适配器必须对空的过期策略报错。
    *   要求适配器必须能返回一个稳定的、可索引的记录ID。
5.  **人机协作治理**：实现了`Co-memorize`差异审批流程，切断了未经审核的恶意写操作直接进入长期存储的通路。

### 7. 优点

*   **方法巧妙**：不发明新算法，而是通过**标准化协议**解决现有生态的碎片化问题，这是非常务实和有效的工程思路。

### 8. 不足与局限

*   **实验数据集规模过小**：微基准测试的100条数据和对抗实验的50条数据都非常小，严重限制了结论的普适性。作者也承认“合成语料库”是其威胁有效性之一。
*   **实验环境非真实生产环境**：所有测试都在开发级笔记本上进行，缺乏分布式部署、网络延迟、真实远程后端（如mem0云服务）的测试。
*   **对抗攻击模型过于简单**：对抗攻击假设了1个完美的恶意后端和2个完美的良性后端，而攻击者也拥有全知（知道router的over-fetch策略）。现实情况可能更复杂（如多个恶意后端、良性后端也可能出现错误）。

*   **治理UI尚未研究**：虽然提出了UI设计，但缺乏用户研究（如NASA-TLX量表）来评估其实际效用和效率。作者已计划在v0.2中弥补这一点。
*   **安全模型未涵盖所有威胁**：作者诚实地列举了协议无法解决的问题，如：**对手对审计日志的写入权限**、**上游LLM的提示注入**（memorywire只能记录但不能防止）、**侧信道时间攻击**。这些是部署前必须考虑的。
