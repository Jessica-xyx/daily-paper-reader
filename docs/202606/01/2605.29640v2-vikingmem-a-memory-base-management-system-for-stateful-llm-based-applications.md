## 摘要
大型语言模型已经彻底改变了交互式应用；然而，其有限的上下文窗口对维护有状态的长期交互构成了关键的数据管理挑战。现有的记忆方法通常依赖于简单的提取方法，导致记忆不完整，或者使用针对单一用例（如聊天机器人）定制的僵化、单一用途的记忆提取提示。因此，它们缺乏泛化能力，在多样的下游任务中表现不佳。为弥补这一差距，我们引入了记忆基础（Memory Base），一种用于管理长期交互持久状态的新型数据管理范式。其特点包括三个核心原则：从原始信息流中选择性提取高价值记忆；固有的状态性和演化性，其中记忆内容被逐步总结、纠正并按时间加权以优先考虑最近交互；以及一种泛化的抽象范式，旨在实现跨不同应用（包括教育、推荐和智能体记忆）的稳健迁移。在此基础上，我们提出了VikingMem，一个在VikingDB向量引擎上实现的端到端记忆基础管理系统。VikingMem通过相互关联的事件和实体抽象来实现这一范式。它利用以事件为中心的记忆提取来选择性处理复杂信息流，同时实体由事件动态更新以实现有状态演化。通过基于主题时间线的时间压缩和时间加权召回，系统逐步生成高层摘要记忆，优先考虑最近项目，并压缩和淡化旧项目。在长期记忆基准上的广泛评估表明，VikingMem在记忆检索效果上优于基线方法高达30%，同时保持交互式应用所需的低延迟。

## Abstract
Large Language Models have revolutionized interactive applications; however, their finite context windows pose a critical data management challenge for maintaining stateful, long-term interactions. Existing memory approaches often rely on simplistic extraction methods that lead to incomplete memories or use rigid, single-purpose memory extraction prompts tailored to a single use case, such as chatbots. Consequently, they lack generalizability and perform poorly across diverse downstream tasks. To bridge this gap, we introduce the Memory Base, a novel data management paradigm for managing the persistent state of long-term interactions. It is characterized by three core principles: selective extraction of high-value memories from raw information streams; inherent statefulness and evolution, where memory content is progressively summarized, corrected, and temporally weighted to prioritize recent interactions; and a generalizable abstraction paradigm designed for robust transferability across diverse applications, including education, recommendation, and agent memory. Building on this foundation, we present VikingMem, an end-to-end Memory Base Management System implemented on the VikingDB vector engine. VikingMem materializes this paradigm through interconnected event and entity abstractions. It features event-centric memory extraction to selectively handle complex information streams, while entities are dynamically updated by events to achieve stateful evolution. Using temporal compression via a topic-wise timeline and time-weighted recall, the system progressively produces high-level summary memories, prioritizes recent items, and compresses and fades older ones. Extensive evaluations on long-term memory benchmarks demonstrate that VikingMem outperformes baselines by up to 30% in memory retrieval effectiveness while maintaining the low latency essential for interactive applications.

### 方法论：核心思想与关键技术

- **核心思想**：将记忆抽象为**事件（Event）**和**实体（Entity）**两大原语。事件是离散、带时间戳的片段记录，实体是持久化、动态演化的状态（如用户画像、智能体SOPs）。事件通过可配置的**操作符（Operator）**聚合生成实体，支持统计操作（SUM/AVG等）和LLM融合（LLM_MERGE、TIME_COMPRESS）。
- **关键技术细节**：
  - **智能记忆分割（IMSM）**：两阶段过程——先滤除低价值内容（如问候语），再通过坐标式分割合并不连续但语义相关的片段，形成完整事件。
  - **单次提取（One-pass Extraction）**：将多个记忆类型的模式编译成单一提示，仅需一次LLM调用即可同时提取所有事件和实体更新内容，大幅降低token消耗（理论上减少约(k-1)倍）。
  - **实体无LLM快速更新（EUA）**：利用编辑距离近似匹配，将LLM输出的“SEARCH/REPLACE”补丁直接应用到旧实体字段，避免额外LLM调用。
  - **多路径检索与动态加权**：混合检索（稠密+稀疏）结合时间衰减权重和业务重要性权重（可配置类型级/实例级），辅以关键词图补充检索。
  - **多向量重排序**：采用ColBERT式延迟交互机制，对候选记忆进行预计算与压缩（量化、token合并），实现高效重排序。
- **算法流程**：输入会话 → 单次LLM提取（生成事件结构化数据+实体补丁） → 去重 → 事件存储 → 操作符驱动实体更新（统计或LLM融合，含时间线压缩） → 查询时多路径召回（主路径+关键词图） → 加权融合 → 多向量重排序 → 输出记忆上下文。

### 主要结论与发现

1. **效果领先**：VikingMem在LOCOMO和LongMemEval上均取得最高LLM-Judge评分，比最优基线（Mirix、Zep）提升约6-25个百分点，尤其在单跳、多跳、开放域问题中优势明显。
2. **效率优异**：P50延迟约0.2-0.25秒，P95延迟低于0.9秒，与轻量方法（Mem0）相当，而高精度方法Mirix延迟高达10秒以上。
3. **存储节约**：相比Naive RAG，存储消耗降低83.18%（压缩比约6:1），且检索精度不降反升（LLM-Judge从63.81提升至75.80）。
4. **组件重要性**：智能记忆分割（IMSM）对质量最敏感（移除后下降5.32分）；重排序以极低延迟（+6.8ms）贡献3.64分提升。
5. **单次提取优势**：相比多提示方法，token成本降低3.2倍，EUA进一步降低延迟和成本，且质量基本持平。

### 优点

- **设计系统性**：从问题剖析到形式化定义（事件-实体模型）再到完整实现，逻辑自洽，覆盖记忆全生命周期（提取、管理、检索）。
- **通用性强**：通过可配置模式和支持多种操作符，适配教育、社交、智能体等不同场景，实验验证了跨场景操作符频率的差异。
- **实用化取向**：针对工业级高吞吐量（单租户日处理1B token）设计，采用缓存、单次提取、无LLM更新等优化，具备实际部署价值。
- **评估全面**：兼顾效果（LLM-Judge、F1）与效率（p50/p95延迟、存储成本），包含消融和组件贡献量化分析。

### 不足与局限

- **实验覆盖**：仅使用两个公开基准（LOCOMO、LongMemEval），未涉及更复杂或更大规模的对话数据集，可能低估长尾场景的挑战。
- **模型依赖**：所有评估基于GPT系列模型，未测试开源模型（如Llama、Qwen）或不同量级模型，泛化结论需谨慎。

- **应用限制**：当前系统依赖VikingDB作为底层存储，未与主流向量库（如Faiss、Milvus）直接比较，迁移性未验证。
