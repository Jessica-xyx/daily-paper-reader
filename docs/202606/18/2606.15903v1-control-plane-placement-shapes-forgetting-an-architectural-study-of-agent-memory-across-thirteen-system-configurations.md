## 摘要
大语言模型在智能体记忆流水线中的位置——即位于检索已存储事实（经过广泛基准测试）的召回平面与通过取代、释放、清除（基本未经测试）来改变这些事实的控制平面之间——决定了系统能够从哪些遗忘失败模式中恢复。通过在385个案例的对抗性表面对比十三种系统配置，我们观察到三种具有部分互补覆盖范围的放置机制：确定性原语足以处理词汇/时间类别，但在规范化上失败（标识符混淆5%，跨语言0%）；写入时的大语言模型可以恢复规范化（100%），但无法帮助意图感知删除（前缀冲突和复合事实均为0%）；突变时钩子恢复了意图感知删除（78-85%），同时几乎照亮了所有类别（总体91.7-93.2%，每次385案例运行成本0.17美元，每个案例突变延迟2.3秒 vs 确定性64-191毫秒/案例，召回路径不变）。我们通过ForgetEval（一个1000案例模板化套件加上385案例对抗层（132个手工制作+253个由大语言模型起草并经预言机验证））暴露了权衡，通过确定性子串匹配评分，并配有一个六方法适配器协议，采用诚实的N/A评分，允许异构记忆存储在130行代码中接入。准入由10名标注者的IAA（Fleiss' kappa = 0.958）和一个77案例的外部作者子集（四名匿名贡献者）证实，该子集复制了规范化不对称性并放大了联合放置提升（+27.8个百分点）。生产故障主要是遗忘故障而非召回故障，但现有基准仅衡量召回。ForgetEval和所有适配器均在MIT许可下发布。

## Abstract
Where an LLM sits in an agent memory pipeline -- between the recall plane that retrieves stored facts (extensively benchmarked) and the control plane that mutates them via supersede, release, purge (largely untested) -- shapes which forgetting failure modes the system recovers. Comparing thirteen system configurations on a 385-case adversarial surface, we observe three placement regimes with partly complementary coverage: deterministic primitives suffice for lexical/temporal categories but fail canonicalization (5% on identifier-obfuscation, 0% on cross-lingual); inscribe-time LLM recovers canonicalization (100%) but cannot help intent-aware deletion (0% on prefix-collision and compound-fact); a mutation-time hook recovers intent-aware deletion (78-85%) and brightens nearly all categories simultaneously (91.7-93.2% overall, $0.17 per 385-case run, 2.3s/case mutation latency vs. 64-191ms/case deterministic, recall path unchanged).   We expose the trade-off via ForgetEval, a 1000-case templated suite plus a 385-case adversarial layer (132 hand-crafted + 253 LLM-drafted oracle-validated) scored by deterministic substring match, paired with a six-method Adapter Protocol with honest N/A scoring that lets heterogeneous memory stores enter in 130 lines. Admission is corroborated by 10-annotator IAA (Fleiss' kappa = 0.958) and a 77-case external-authored subset (four blind contributors) that replicates the canonicalization asymmetry and amplifies the joint-placement lift (+27.8 pt). Production failures are predominantly forgetting failures rather than recall failures, yet existing benchmarks measure only recall. ForgetEval and all adapters are released under MIT.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：当前LLM智能体记忆系统的评测基准（如LongMemEval、MTEB、BEIR）几乎全部聚焦于**召回能力**（recall），即“检索到正确事实”。然而，实际生产中的主要故障是**遗忘故障**（forgetting），例如：已轮换的密码仍被推荐、GDPR删除记录仍出现在候选池、用户职位存在多个矛盾版本等。这些故障源于**控制平面**（control-plane）缺失——系统无法正确执行“遗忘”操作（如取代、释放、清除）。
- **核心问题**：LLM在记忆管道中的**放置位置**（placement）——位于召回平面（检索已存储事实）与控制平面（通过superdede/release/purge对内存进行变更）之间——如何影响系统对各类遗忘失败模式的恢复能力？不同放置机制（确定性原语、写入时LLM、变异时hook）存在怎样的权衡？
- **整体含义**：论文首次系统定义了遗忘失败的五种结构类型，并揭示了**架构放置**（而非单纯的有无LLM）决定遗忘恢复模式。研究为生产级记忆系统设计提供了重要的实证依据，提出了一个标准化的遗忘评测方法（ForgetEval）和可扩展的Adapter协议。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：通过将LLM放置在记忆管道的不同关键位置（写入时、变异时），对比不同架构对10种对抗性攻击类别的恢复能力。识别三种互补的放置机制，并揭示了“联合放置”（同时使用写入时LLM和变异时hook）的必要性。
- **关键技术细节**：
    - **五大家族遗忘原语**：Supersession（新事实覆盖旧事实）、Decay（TTL释放）、Amnesia（忘记一个实体的所有事实）、Purge（按标识符硬删除）、Drift（链式取代且中间态不可达）。
    - **评估指标**：确定性**子串匹配**（must_contain ⊆ top-10，must_not_contain ∩ top-10 = ∅），不使用LLM做裁判。
    - **Adapter协议**：强制6个方法（reset、inscribe、recall_texts + 可选supersede、release、purge），缺失的方法按N/A（并非失败）处理。该协议允许异构系统（向量库、知识图谱、SQLite等）在~130行代码内接入。
    - **引用存储（Lethe）**：基于SQLite+sqlite-vec+FTS5实现，支持深度标量（depth）和软删除、审计日志。
    - **LLM钩子**：在变异时调用LLM（DeepSeek-V3）进行JSON格式的“取代规划器/清除匹配器”生成，召回路径完全无LLM。

## 6. 主要结论与发现

- **三种放置机制互补但不替代**：
    - **确定性原语**（无LLM）：在词汇/时间类攻击（子串陷阱、否定、时间限定）上表现优异（≥92%），但在规范化类（标识符混淆5%、跨语言0%）几乎失败，在意图感知删除类（前缀碰撞31-82%）依赖精确匹配。
    - **写入时LLM**（如A-MEM、Mem0+v3）：在**规范化**（标识符混淆100%、跨语言100%）完全恢复，但**完全无法处理**前缀碰撞、复合事实等意图感知删除（0%），因为其倾向于“链接并保留”而不执行精确删除。
    - **突变时hook**（如Lethe+LLM、LangGraph+LLM）：恢复所有意图感知删除类别（前缀碰撞79%、复合事实78-85%），同时提升全部非基础类别（排除复合事实后345例中总体93.3-94.2%，提升22-24个百分点）；延迟可接受（2.3秒/例，成本0.17美元/385例），召回路径不变。
- **联合放置（写入时LLM+变异时hook）进一步提升**：Letta+LLM (写入时LLM+变异时hook) 在外部子集上相比仅Letta提升27.8个百分点，且超过仅变异时hook的Lethe+LLM/LangGraph+LLM，显示必要的联合放置。
- **记忆目标对比**：
    - MemPalace（无遗忘原语）在ForgetEval-Adv上0/385，但在召回密集的Memora上有40%通过率——验证了两个维度不相关。
    - Lethe 在ForgetEval-Adv（遗忘）得分63.4%，在Memora（召回）得分31%，而LangGraph在遗忘上62.9%、回忆上45%；存在明显的召回-遗忘权衡。
- **控制平面空缺**：63-68%的确定性通过带意味着约1/3的精确删除泄漏。LLM钩子可将其提升至92-93%，但深度依赖LLM的JSON遵从能力。

## 7. 优点

- **问题定义创新**：首次系统地将遗忘失败分解为5个家族和10个攻击类别，弥补了现有记忆基准仅关注召回的空缺。
- **评测方法严谨**：采用**确定性子串评分**（无需LLM裁判），提高了可重复性和公平性；设置LLM法官仅用于准入阶段，并通过10人IAA和跨家族法官验证，避免“循环论证”问题。
- **架构泛化**：Adapter协议允许异构记忆系统在130行代码内接入，公平比较（N/A标记缺失功能），显著降低未来研究的入门门槛。
- **实证覆盖全面**：在13种系统配置、6种放置机制、385+1000例上进行了系统化对比，并加入外部作者验证和跨消融（LLM、嵌入器、架构），增强了结论的稳健性。
- **开放和可复现性**：代码、数据、协议全部以MIT许可发布，并提供了单命令生成所有图表的能力。

## 8. 不足与局限

- **语言与脚本覆盖不足**：主要测试为英语，跨语言类别只有9种族裔，且每个族样本量极薄（部分仅1-3例），因此泰语、阿拉伯语、Hirag等的结果统计显著性不足。
- **记忆系统覆盖有限**：主要对比了4个开源系统（Lethe, Mem0, LangGraph, MemPalace）；其他系统（A-MEM, Letta, Graphiti, Hip-pRAG, OpenMemory）虽已评估但属于扩展组，且存在一些API不兼容（如Graphiti失配LLM抽象层导致N/A率高，Mem0的infer=True模式下JSON解析问题等）。Redis Semantic Memory、AutioGen等未评估。
- **复合事实类的语义歧义**：该类别本质上是部分编辑能力测试（非纯遗忘），标注者与LLM裁判之间存在分歧（10/21分歧集中在此）。虽然作者已特别标注，但可能影响某些聚合解读。
