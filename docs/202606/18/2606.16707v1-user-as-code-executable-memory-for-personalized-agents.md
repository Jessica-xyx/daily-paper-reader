## 摘要
个性化AI代理需要用户记忆：一种持久的用户模型，跨多次对话构建，并在每次新对话中参考。如今，这种记忆几乎总是以非结构化文本、知识图谱或扁平的事实存储形式存储，并通过检索来获取——即获取与当前请求最相似的条目。这种“事实袋”记忆能够很好地回忆起个别事实，但由于存储事实和执行操作是分开的步骤，它在解决矛盾、聚合多条记录或执行规则方面存在困难。我们认为用户记忆应该是可执行的。我们提出了“用户即代码”（UaC）范式，其中代理对用户的模型是一个活的软件项目：类型化的Python对象保存用户状态，普通的Python函数编码管理状态的规则，因此在一个解释器可以运行的媒介中同时表示和推理用户。实现机制是一个两阶段流水线：一个仅追加的日志，从不丢弃任何事实，定期检查点化为类型化代码。这改变了记忆的功能。在标准的长期对话基准测试中，UaC在全上下文上限和最强的先前记忆系统上均匹配召回率（LOCOMO上为78.8%）。其优势在表示至关重要的地方显现。对于用户历史的聚合问题——“我去年进行了多少次国际旅行？”——基于检索的记忆崩溃（6-43%），而UaC几乎完美（99%），因为答案是对类型化状态进行单行计算，而不是对文本进行搜索。而且，由于其规则在状态变化时确定性执行，UaC可以呈现主动的、安全关键的警报——例如，新处方的药物与几个月前记录的过敏冲突——这是查询驱动记忆无法提供的能力。

## Abstract
A personalized AI agent needs a user memory: a persistent model of who the user is, built across many conversations and consulted on each new one. Today this memory is almost always stored as unstructured text, a knowledge graph, or a flat store of facts, and consulted by retrieval -- fetching the entries most similar to the current request. Such "bag-of-facts" memory recalls individual facts well, but because storing a fact and acting on it are separate steps, it struggles to resolve contradictions, aggregate over many records, or enforce rules. We argue that user memory should instead be executable. We introduce User as Code (UaC), a paradigm in which an agent's model of a user is a living software project: typed Python objects hold the user's state and ordinary Python functions encode the rules that govern it, so representing and reasoning about the user happen in one medium an interpreter can run. The enabling mechanism is a two-phase pipeline: an append-only log that never discards a fact, periodically checkpointed into typed code.   This changes what memory can do. On standard long-term conversation benchmarks, UaC matches both a full-context upper bound and the strongest prior memory systems on recall (78.8% on LOCOMO). Its advantage emerges where representation matters most. On aggregate questions over a user's history -- "how many international trips did I take last year?" -- retrieval-based memory collapses (6-43%) while UaC stays near-perfect (99%), because the answer is a one-line computation over typed state rather than a search over text. And because its rules execute deterministically whenever the state changes, UaC can surface unsolicited, safety-critical alerts -- such as a newly prescribed drug that conflicts with an allergy recorded months earlier -- a capability query-driven memory cannot provide.

## 1. 核心问题与整体含义（研究动机和背景）
当前个性化AI代理的用户记忆通常以非结构化文本、知识图谱或扁平的事实存储（“事实袋”）形式存在，并依赖检索（如Top-k向量相似度）来召回信息。这种设计存在三个根本缺陷：
- **无法处理矛盾**：如“我爱香菜”和“我讨厌香菜”两个矛盾事实共存，无版本历史或分辨机制。
- **无法聚合**：对于“我去年进行了多少次国际旅行？”等需要统计全体记录的问题，检索只能返回少数片段，导致准确率急剧下降（6-43%）。
- **无法执行规则**：非结构化格式无法表达如“若护照在出发前180天内过期则发出警报”之类的可执行约束，无法主动触发安全关键告警。

本文的核心动机是**将用户记忆从被动的、查询驱动的检索式记忆，转变为主动的、可执行的代码式记忆**。其整体含义是：通过使用类型化Python对象和函数来统一表示与推理，使代理能够确定性计算、聚合查询和无人询问的主动服务，从而从根本上提升个性化代理的能力边界。

## 2. 方法论（核心思想、关键技术细节、算法流程）
### 核心思想
将用户的个人模型视为一个**活着的软件项目**：状态由类型化Python dataclass实例表示，规则由普通Python函数编码。这样，解释器可以直接读取和计算，无需LLM每次重新解析文本。

### 关键技术细节
采用**两阶段流水线**（受数据库系统启发）：
- **Phase 1: Memorize（追加日志）**  
  每会话使用LLM（Gemini 3 Flash，思考预算8192）提取所有事实为扁平字符串，追加到运行列表（从不覆盖、从不删除），并将相对日期解析为绝对日期。事实数约50-75/会话。结果存入ChromaDB（事实向量索引+原始对话存档索引）。
- **Phase 2: Structure（定期检查点）**  
  从完整事实集中使用LLM（思考预算16384）重新生成整个结构化Python代码：自动设计dataclass、日期类型化、分组集合、notes字段承接难以类型化的事实。代码完全由LLM生成，无人工预定义模式。

### 检索与使用（双模式）
- **多策略检索**：组合三个通道——结构化代码状态（state.py，截断至6000字符）、事实向量Top-20、原始对话存档Top-10，直供给回答LLM。
- **主动服务（Generate-Verify-Review Loop）**：  
  1. 编码代理编写Python约束函数（如检查药物过敏冲突）。  
  2. 沙箱执行，结果确定性。  
  3. 若有用则提升为持续约束，每次状态变化时自动执行，结果写入清单（manifest）的`ACTIVE_ALERTS`，在每会话开始处加载（约300token）。

### 设计原则
- 结构与数据分离（schema vs. instance）。
- 统一表示与验证（代码既是存储也是验证工具）。
- 模块化（按生活领域划分独立包）。
- 渐进披露（紧凑清单 + 按需加载领域）。
- 代理原生文件系统（无需自定义API）。

3. **主动告警能力首次实现**：在标准Active Service上UaC+约束流水线达100%，无流水线时仅52.5%。在hard场景上UaC 85%，优于所有基线（Full Context 55%, Mem0 80%但统计不显著）。
4. **消融关键发现**：追加日志（+19pp）和两阶段分离（+12.3pp over增量代码）是性能核心。结构化代码对性能召回贡献不显著（-1.3pp, p=0.67），但支撑了聚合和主动服务。
5. **成本摊销优势**：结构化一次性成本在3-11次查询后回本，在100次查询场景下UaC比Full Context+REPL便宜15.5倍。
6. **跨模型可迁移**：GPT-5.4替代整个流水线后性能持平（80.8% vs 82.5%），Claude Opus 4.7重评保持排名（Cohen’s κ≥0.74）。

## 8. 不足与局限
### 实验覆盖
- **Hard Active Service统计不足**：20个场景下UaC与Mem0的差异未达显著水平，需要更大规模（~200场景）验证。
- **合成数据偏差**：Analytical和Modularity实验使用干净结构化数据，真实对话噪声（如歧义、缺失信息）可能不利于UaC的代码生成质量。
- **缩放上限未充分解决**：Phase 2一次性再生在超过100会话时输入被截断，论文仅提出未来方向，无实际分层方案。

### 偏差风险
- **同家族评分器偏差**：虽然经过Claude重评验证（κ≥0.74-0.97），但主要实验仍使用Gemini-as-Judge，可能存在系统间隐式偏好（但重评显示排名不变且UaC优势扩大）。
- **基线再实现偏差**：SOTA基线（MemMachine等）的再实现使用但弱于原版骨干和存储栈，可能导致低估它们的上限能力。论文明确标注了这一差异。

### 应用限制
- **写入开销**：每会话需要两次LLM调用（Phase 1+Phase 2），延迟较高（UaC回答中位3.62秒，约2× Full Context），适合后台异步处理而非实时交互。
- **依赖LLM代码生成质量**：Phase 2和约束生成依赖LLM能力，较小模型可能质量下降。
- **沙箱需求**：可执行代码需要严格沙箱隔离以防止注入或跨用户泄露，增加工程复杂度。
- **结构化代码对纯召回贡献小**：对于仅需单跳事实检索的场景，UaC的额外开销可能不值得。

### 其他
- 未考虑跨语言对话、隐私法律约束（如GDPR遗忘权）等实际部署问题。
- 约束逻辑的正确性依赖LLM生成，且确定性执行意味着错误约束会重复触发（需要人工校验机制）。
