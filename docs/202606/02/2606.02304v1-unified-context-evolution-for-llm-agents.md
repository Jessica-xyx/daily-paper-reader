## 摘要
基于LLM的智能体通过结合推理与环境反馈解决多步交互任务，但每一轮都从相同的固定上下文开始，过程中发现的任何有用策略一旦任务结束便丢失。现有方法要么将学习限制在当前任务，要么将所有经验汇集到单一无类型存储中，既不区分知识类型，也不通过使用追踪质量，更未平衡知识库尚缺的内容。我们提出统一上下文演化（UCE）——一种无梯度框架，将智能体经验外部化为包含类型化可演化上下文单元（ECU）的演化库。UCE将经验分解为四种互补类型（记忆、策略、工作流和技能），每种类型从轨迹中在特定类型条件下生成，在决策时检索，通过重复使用结果评分，并在不再有价值时剪枝。调度模块将每轮生成预算分配给库中最弱的类型。在两个交互基准测试中，UCE将ALFWorld成功率从75.4%提升至96.3%，WebShop任务得分从45.1%提升至61.3%，并且累积的库无需重新训练即可迁移至其他智能体骨干网络。

## Abstract
LLM-based agents can solve multi-step interactive tasks by combining reasoning with environment feedback, yet each episode starts from the same fixed context and any useful strategy discovered along the way is lost once the task ends. Existing approaches either limit learning to the current task or pool all experience into a single untyped store, without distinguishing knowledge types, tracking quality through use, or balancing what the library still lacks. We introduce Unified Context Evolution (UCE), a gradient-free framework that externalizes agent experience into an evolving library of typed Evolvable Context Units (ECUs). UCE decomposes experience into four complementary types (Memory, Strategy, Workflow, and Skill), each generated from trajectories under type-specific conditions, retrieved at decision time, scored through repeated usage outcomes, and pruned when no longer valuable. A scheduling module allocates each cycle's generation budget toward the types where the library is weakest. Across two interactive benchmarks, UCE raises ALFWorld success from 75.4% to 96.3% and WebShop task score from 45.1% to 61.3%, and the accumulated library transfers to alternative actor backbones without retraining.

## 一、核心问题与整体含义（研究动机和背景）

- **核心问题**：当前基于LLM的交互式智能体在每轮任务中从一个固定的上下文（系统指令、少样本示例、工具模式等）开始执行，在任务过程中发现的任何有用策略（如环境机制、决策规则、工作流程）在任务结束后全部丢失。智能体无法跨任务积累和复用经验，每个新任务都从零开始。
- **背景意义**：现有方法要么局限于当前任务内自我调整（如Reflexion），要么将所有经验混入单一无类型的存储中，既不区分知识类型，也不通过实际使用追踪质量，也未考虑库中缺失了什么。这限制了智能体在持续交互环境中的学习效率。
- **整体含义**：论文主张让上下文（context）本身成为学习的场所，通过将经验外化为结构化的、可演化的知识库，实现无梯度的持续改进。

## 二、论文提出的方法论

### 核心思想
- 提出**统一上下文演化（Unified Context Evolution, UCE）**框架，维护一个可演化的**可演化上下文单元（Evolvable Context Units, ECUs）**库。
- 每个ECU是一个五元组：`(类型, 任务类型, 检索条件, 知识内容, 适应度分数)`。
- 将智能体经验分解为四种**互补类型**：
  1. **Memory（记忆）**：环境机制的事实现象（如“水槽盆无需打开水龙头即可清洗物品”）。
  2. **Strategy（策略）**：特定条件下的决策规则（IF-THEN形式）。
  3. **Workflow（工作流）**：任务类型的标准执行序列。
  4. **Skill（技能）**：可跨任务类型复用的多步子操作方法。

### 关键技术细节
- **生成条件**：每种ECU有独立的生成条件。Memory从单条轨迹提取；Strategy从同任务类型多条轨迹提取；Workflow从同任务类型成功轨迹提取；Skill从跨任务类型轨迹提取。
- **检索机制**：执行任务前，检索器对每个ECU评分：`score = 语义相似度 + β·类型匹配 + λ·(适应度分数-0.5)`。每种类型独立Top-K注入到prompt，防止单一类型主导。
- **适应度分数**：`f = (s+1)/(u+2)`，其中s是累积成功信用，u是使用次数。开始为0.5，随使用更新。还维护任务类型条件分数。
- **演化周期（5个阶段）**：
  1. **Evaluate**：在评估集上运行智能体，记录成功率（不更新适应度）。
  2. **Collect**：在采集集上运行，存储轨迹和奖励，更新适应度分数。
  3. **Schedule（KYS）**：知识产量调度，根据库状态计算每种类型的生成预算。
  4. **Generate**：对每种类型使用特定提示从收集轨迹中提取ECU候选，经去重后加入库。
  5. **Cleanup**：移除低质量或冗余ECU。

### 知识产量调度（KYS）细节
- **三层结构**：
  - 第一层：可行性过滤——检查每种类型是否还有生成空间（是否存在无足够ECU覆盖的任务类型）。
  - 第二层：需求评估——计算每种类型的知识缺口，Memory和Strategy结合覆盖缺口与质量缺口，Workflow仅覆盖缺口，Skill参考绝对数量。
  - 第三层：产量追踪——使用指数移动平均（EMA）跟踪每种类型的历史回报。
- **分数融合**：`s = α·demand + (1−α)·yield`，α随周期衰减。标准化后分配最低份额，其余按比例分配预算。

## 三、实验设计

### 数据集与场景
- **ALFWorld**：134个家务任务，6种任务类型（Pick, Look, Clean, Heat, Cool, Pick2）。文本交互环境，离散动作。
- **WebShop**：100个在线购物任务，5个产品子类别（Beauty, Clothing, Electronics, Food, Home）。文本网页交互，连续奖励评分（0~1）。

### Benchmark对比方法
- **基线**：ReAct（基本循环）、Reflexion（带自反思的多试）、ExpeL（体验学习，提取自然语言洞察）。
- **替代骨干**：NoThinking（无推理）、Plan-and-Act（仅首步计划）、ReflAct（每步反思）。
- **所有方法**：使用相同的actor模型（gpt-4.1-mini）、生成器/反思模型（gpt-5.2）、任务集和步数预算。

### 评估协议
- ALFWorld运行5个周期，WebShop运行11个周期。
- 周期0为ReAct基线，每个任务每周期仅评估一次，无多次尝试。
- 使用k=2分层分割，保持训练/评估集独立。

## 六、论文的主要结论与发现

1. **UCE显著提升性能**：ALFWorld成功率从75.4%提升至96.3%（+20.9pp）；WebShop任务得分从45.1%提升至61.3%（+16.2pp），购买成功率从30.0%提升至42.0%。
2. **库可迁移**：累积的ECU库可以零额外训练地迁移到NoThinking、Plan-and-Act、ReflAct等不同actor骨干，并带来可观提升。
3. **不同类型各有贡献**：ALFWorld中Workflow单独就接近全库性能（97.0% vs 96.3%），但WebShop需要多种类型协同，单类型无法匹配全库。
4. **知识调度有效**：KYS自适应分配预算，使库在早期快速提升，后续稳定。
5. **错误模式**：WebShop主要失败源自属性匹配错误（50%）和步数超时（27%），表明UCE擅长环境机制和过程知识，但精确属性选择仍需强化学习等策略优化方法。

## 七、优点

1. **梯度自由**：不更新模型参数，仅通过外部上下文进化，适用于闭源API模型。
2. **类型化知识**：四种类型各司其职，互补协同，避免单一知识形式的局限。
3. **自适应调度**：KYS基于库状态动态分配生成预算，避免饱和类型浪费资源，优先补足薄弱环节。
4. **实用健壮性**：通过使用次数和结果计算适应度分数，支持剪枝和去重，库随使用自动优化。
5. **跨骨干迁移**：知识独立于特定actor提示格式，可即插即用。
6. **实验设计规范**：严格控制评估集不与采集集重叠，适应度更新仅来自采集轨迹，确保评估信号纯洁性。

## 八、不足与局限

1. **环境覆盖有限**：仅在两个文本交互环境（ALFWorld和WebShop）上验证，缺乏连续动作空间、多模态输入（如网页截图）、更长规划场景（如AndroidWorld、VisualWebArena）的测试。
2. **模型泛化风险**：使用单一模型对（gpt-4.1-mini作为智能体，gpt-5.2作为生成器），无法区分性能提升来自框架设计、生成器蒸馏能力还是环境规律性。
3. **任务池规模小**：ALFWorld 134个任务，WebShop 100个任务，调度信号可能不稳定；大规模任务集或长演化周期下库是否会退化尚未验证。
4. **类型设计手工艺化**：四种类型目前是手工设计，缺乏自动类型发现机制；不同领域可能需要不同分类。
5. **失败模式边界**：属性精确匹配等精细化操作超出UCE能力范围，仍需与强化学习方法结合。
6. **计算成本**：虽然无梯度，但每次演化周期需要调用昂贵生成器（gpt-5.2）提取ECU，且WebShop服务器需大量内存。
